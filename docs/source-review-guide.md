# Source Review Guide

This guide is for a judge who wants to inspect the technical design. It maps the main areas of the codebase and explains what to look for in each one.

The goal is ten minutes of focused reading, not coverage of every file.

---

## System Flow

```
Query
  → Retrieval (similarity)
  → Context Selection
  → LLM Generation
  → Deterministic Evaluation         ← scores the answer against a rubric
  → Query Expansion (from missing facts)
  → Retrieval (Run 2, expanded)
  → Improved Answer
```

---

## Where EvoContext Differs From a Typical RAG Pipeline

Most RAG systems stop after the first generation. EvoContext adds a feedback loop. The key differences:

1. **Deterministic rubric evaluation** — the answer is scored by rule-based fact matching, not by a model judge. The same answer always receives the same score.
2. **Fact grounding via context anchors** — a fact is only marked present if both the answer text and the retrieved context contain the expected phrases. This prevents scoring points for hallucinated content.
3. **Query expansion derived from missing fact labels** — the Run 2 queries are looked up from a static table keyed on missing fact codes, not generated by the model. Expansion is traceable and deterministic.
4. **Structured trace artifacts** — every run produces a JSON artifact capturing both retrieval passes, scores, and evaluator reasoning. Replay is possible without any external services.

## Repo Map

```
src/EvoContext.Core/           Contracts, domain models, rubric, evaluation logic
src/EvoContext.Infrastructure/ External service implementations and orchestration
src/EvoContext.Cli/            CLI host — operator-facing commands
src/EvoContext.Demo/           Demo host — judge-facing narrated output
data/scenarios/                Scenario document datasets
validation/                    Gate A (retrieval precision), Gate B (hallucination stress test)
tests/                         Unit and regression tests
```

The split between `Core` and `Infrastructure` is deliberate: evaluation logic, rubric rules, and domain contracts live in `Core` with no external dependencies. OpenAI calls, Qdrant calls, and orchestration live in `Infrastructure`.

## Entry Points

There are two runnable hosts. Both share the same underlying pipeline.

| Host | File | Purpose |
|---|---|---|
| CLI | [src/EvoContext.Cli/Program.cs](src/EvoContext.Cli/Program.cs) | Operator-facing. Raw evaluator codes, structured key/value output. |
| Demo | [src/EvoContext.Demo/Program.cs](src/EvoContext.Demo/Program.cs) | Judge-facing. Same pipeline, narrated output with human-readable labels. |

Both dispatch commands the same way: `args[0]` selects the command, the rest are passed to the handler. The demo host wraps the CLI facade with a renderer that maps internal codes to plain English.

## CLI Commands

| Command | What it does |
|---|---|
| `embed` | Loads documents, generates embeddings via OpenAI, stores vectors in Qdrant. Runs Gate A on completion. |
| `run --mode run2` | Executes the full adaptive pipeline: Run 1 → evaluate → Run 2 if score is below threshold. |
| `run --mode run1` | Single-pass only. No adaptive retry. Useful for baseline comparison. |
| `replay --run-id` | Reads a trace artifact from disk and re-renders it. No external calls. |
| `ingest` | Dry-run document load only. No embedding or storage. |
| `stats` | Aggregate stats across stored trace artifacts for a scenario. |
| `config` | Prints all resolved configuration values. Secrets shown as `***`. |

Commands are defined in [src/EvoContext.Cli/Program.cs](src/EvoContext.Cli/Program.cs). Each delegates to a service in `src/EvoContext.Infrastructure/Services/`.

## Scenario Loading

Scenarios are directory-based. A scenario ID like `policy_refund_v1` resolves to `data/scenarios/policy_refund_v1/`. The directory contains the policy documents as `.md` files.

Loading path: `ScenarioLoader` → `MarkdownDocumentLoader` → `DocumentIngestionService` → `DocumentChunking`

- [src/EvoContext.Infrastructure/Services/ScenarioLoader.cs](src/EvoContext.Infrastructure/Services/ScenarioLoader.cs) — resolves scenario directory and loads documents
- [src/EvoContext.Core/Documents/DocumentChunking.cs](src/EvoContext.Core/Documents/DocumentChunking.cs) — fixed-size chunking (1200 char size, 200 char overlap). Parameters are pinned constants, not runtime config.
- Document IDs are parsed from the filename prefix (`01_*.md` → doc ID `"01"`). Chunk IDs are `{docId}_{chunkIndex}`.

**What to notice:** chunking is deterministic and locked. There is no ML-based chunking or dynamic splitting.

## Retrieval Pipeline

The retrieval pipeline converts a query to a vector, searches Qdrant, and returns ranked candidates.

- [src/EvoContext.Infrastructure/Services/EmbeddingService.cs](src/EvoContext.Infrastructure/Services/EmbeddingService.cs) — calls OpenAI `text-embedding-3-small` to embed the query
- [src/EvoContext.Infrastructure/Services/RetrievalService.cs](src/EvoContext.Infrastructure/Services/RetrievalService.cs) — searches Qdrant with cosine distance, returns top N=10 candidates
- [src/EvoContext.Infrastructure/Services/CandidateScorer.cs](src/EvoContext.Infrastructure/Services/CandidateScorer.cs) — applies a secondary scoring pass on top of the similarity score
- [src/EvoContext.Infrastructure/Services/CandidateRanker.cs](src/EvoContext.Infrastructure/Services/CandidateRanker.cs) — ranks the scored candidates
- [src/EvoContext.Infrastructure/Services/CandidatePoolMerger.cs](src/EvoContext.Infrastructure/Services/CandidatePoolMerger.cs) — in Run 2, merges the candidate pools from all expansion queries into a single deduplicated pool

**What to notice:** in Run 2 `retrieval_queries` in the trace will contain the original query plus the expansion queries. The merger deduplicates by chunk ID so each chunk appears once regardless of how many queries retrieved it.

---

## Context Selection

After retrieval, 3 chunks are selected from the top N candidates and packed into the context window.

- [src/EvoContext.Infrastructure/Services/ContextSelector.cs](src/EvoContext.Infrastructure/Services/ContextSelector.cs) — selects K=3 chunks from the ranked pool
- [src/EvoContext.Infrastructure/Services/ContextPackPacker.cs](src/EvoContext.Infrastructure/Services/ContextPackPacker.cs) — packs selected chunks into a single context string, respecting a 2200-char budget

**What to notice:** the context budget is a locked constant, not a parameter. Selection K=3 is also locked. These are not tunable at runtime.

## Prompt and Generation Layer

The answer is generated from a structured prompt that includes the query and the packed context.

- [src/EvoContext.Infrastructure/Services/Phase3PromptBuilder.cs](src/EvoContext.Infrastructure/Services/Phase3PromptBuilder.cs) — builds the prompt for the policy refund scenario
- [src/EvoContext.Infrastructure/Services/Runbook502PromptTemplate.cs](src/EvoContext.Infrastructure/Services/Runbook502PromptTemplate.cs) — prompt template for the runbook scenario
- [src/EvoContext.Infrastructure/Services/AnswerGenerationService.cs](src/EvoContext.Infrastructure/Services/AnswerGenerationService.cs) — calls `gpt-4.1` with temperature=0, max_tokens=350. Temperature 0 makes output as deterministic as possible.
- [src/EvoContext.Infrastructure/Services/AnswerFormatValidator.cs](src/EvoContext.Infrastructure/Services/AnswerFormatValidator.cs) — validates structural format of the answer before passing it to the evaluator

**What to notice:** temperature=0 is a locked constant. The prompts instruct the model to respond in a structured format (sections A–E for policy, numbered steps for runbook). The format validator enforces this before evaluation runs.

## Evaluator and Rubric

This is the core of EvoContext's value proposition. The evaluator is deterministic and rule-based — no model judge.

**Dispatcher:**
- [src/EvoContext.Core/Evaluation/ScenarioEvaluatorDispatcher.cs](src/EvoContext.Core/Evaluation/ScenarioEvaluatorDispatcher.cs) — routes evaluation to the correct scenario evaluator based on scenario ID

**Policy refund evaluator:**
- [src/EvoContext.Core/Evaluation/PolicyRefundEvaluator.cs](src/EvoContext.Core/Evaluation/PolicyRefundEvaluator.cs) — orchestrates fact checking, hallucination detection, and scoring for `policy_refund_v1`
- [src/EvoContext.Core/Evaluation/Phase4FactEvaluator.cs](src/EvoContext.Core/Evaluation/Phase4FactEvaluator.cs) — checks each fact rule against the answer
- [src/EvoContext.Core/Evaluation/Phase4HallucinationDetector.cs](src/EvoContext.Core/Evaluation/Phase4HallucinationDetector.cs) — detects fabricated claims in the answer
- [src/EvoContext.Core/Evaluation/Phase4ScoreCalculator.cs](src/EvoContext.Core/Evaluation/Phase4ScoreCalculator.cs) — computes the final score from completeness points, format points, and hallucination penalty

**Runbook evaluator:** parallel structure under `Runbook502*.cs` files. Checks for required steps and step ordering rather than facts.

**The rubric:**
- [src/EvoContext.Core/Evaluation/Phase4RuleTables.cs](src/EvoContext.Core/Evaluation/Phase4RuleTables.cs) — the complete policy refund rubric. Contains:
  - 5 `FactRule` entries (F1–F5), each specifying: answer patterns that must be present, context anchor phrases that confirm grounding, and optional negation guards
  - 4 `HallucinationRule` entries for fabricated claims
  - `QuerySuggestions` — the expansion query strings generated for each missing fact label. **This is where Run 2 query expansion comes from.**

**What to notice in the rubric:**
- Each fact rule has both `AnswerPatterns` (what must appear in the answer) and `ContextAnchors` (what must exist in the retrieved context). A fact is only marked present if both conditions are met — this prevents the model from scoring points by hallucinating a fact that was never in the context.
- `RequiresDualAnswerMatch` on F2 and F5 means two independent answer patterns must both match. This reduces false positives for complex facts.
- `NegationGuardPatterns` on F2 catches cases where the model says "no prorated refund mentioned" — which should not score as present.

## Feedback Extraction and Run 2 Expansion

The link between evaluation output and Run 2 query construction is explicit and traceable.

- [src/EvoContext.Core/Evaluation/Phase4QuerySuggestionMapper.cs](src/EvoContext.Core/Evaluation/Phase4QuerySuggestionMapper.cs) — maps each missing fact label to a list of targeted query strings using the `QuerySuggestions` table in `Phase4RuleTables`
- [src/EvoContext.Infrastructure/Services/Run2QueryBuilder.cs](src/EvoContext.Infrastructure/Services/Run2QueryBuilder.cs) — assembles the full Run 2 query list: original query + expansion queries
- [src/EvoContext.Infrastructure/Services/Run2CandidateScorer.cs](src/EvoContext.Infrastructure/Services/Run2CandidateScorer.cs) — applies a Run 2-specific scoring pass that favours candidates retrieved by expansion queries

**Trace of the feedback loop:**

```
Evaluator produces: [MISSING_ANNUAL_PRORATION_RULE, MISSING_BILLING_ERROR_EXCEPTION, ...]
        ↓
Phase4QuerySuggestionMapper produces:
  ["early termination prorated reimbursement",
   "service commitment contract year unused service value",
   "duplicate charge billing error refund", ...]
        ↓
Run2QueryBuilder prepends original query → full query list
        ↓
RetrievalService called once per expansion query
        ↓
CandidatePoolMerger deduplicates and merges
        ↓
ContextSelector picks top K=3 from merged pool
```

**What to notice:** the expansion queries are not generated by the model. They are looked up from a static table keyed on fact labels. This makes the expansion step deterministic and auditable.

## Trace Persistence

Every run emits structured trace events. These are written to a JSON artifact after the run completes.

- [src/EvoContext.Core/Tracing/TraceEvent.cs](src/EvoContext.Core/Tracing/TraceEvent.cs) — the event record: type, run ID, sequence index, metadata dictionary, timestamp
- [src/EvoContext.Infrastructure/Services/TraceEmitter.cs](src/EvoContext.Infrastructure/Services/TraceEmitter.cs) — buffers events during the run
- [src/EvoContext.Infrastructure/Services/TraceArtifactWriter.cs](src/EvoContext.Infrastructure/Services/TraceArtifactWriter.cs) — serializes the run to `artifacts/traces/{scenario_id}/{run_id}.json`
- [src/EvoContext.Infrastructure/Services/RunVerificationEvidenceWriter.cs](src/EvoContext.Infrastructure/Services/RunVerificationEvidenceWriter.cs) — writes the `.verification.json` snapshot of Run 1 state before Run 2 executes

Event types in order during a run: `RunStarted` → `RetrievalCompleted` → `ContextSelected` → `GenerationCompleted` → `EvaluationCompleted` → `Run2Triggered` (if applicable) → second retrieval/generation/evaluation cycle → `RunFinished` → `RunSummary`

**What to notice:** the trace is not a log. It is a structured artifact with typed events and metadata dictionaries. The `SequenceIndex` field orders events deterministically. This is what makes replay possible.

---

## Replay Path

Replay re-renders a stored trace without calling any external service.

- [src/EvoContext.Infrastructure/Services/TraceArtifactReader.cs](src/EvoContext.Infrastructure/Services/TraceArtifactReader.cs) — reads `{run_id}.json` from `artifacts/traces/{scenario_id}/`
- [src/EvoContext.Demo/DemoRunRenderer.cs](src/EvoContext.Demo/DemoRunRenderer.cs) — consumes trace events and renders the narrated summary. The same renderer is used for both live runs and replay.

**What to notice:** the renderer is stateless with respect to external services. Given the same trace events, it always produces the same output. This is why replay is a reliable demo path.

## Validation Gates

These are not tests — they are pipeline integrity checks that run against the live system.

- [validation/EvoContext.Validation.GateA/](validation/EvoContext.Validation.GateA/) — Gate A: runs the refund query against the live Qdrant collection and asserts that Doc 06 (early termination policy) does NOT appear in the top K=3 results. This validates that similarity-only retrieval has the known limitation — Doc 06 is semantically close but not the right document. Exit `0` = pass, `2` = fail.
- [validation/EvoContext.Validation.GateB/](validation/EvoContext.Validation.GateB/) — Gate B: runs 20 identical generation passes with a locked context and checks for hallucinated content. Exit `0` = pass, `2` = one or more hallucinations detected.

Gate A is also embedded in the `embed` command — it runs automatically after dataset setup.
